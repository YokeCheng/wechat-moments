# 2026-04-13 本轮交付审计

## 1. 审计目的

本审计用于把本轮改动重新拉回 Harness 约束内，回答三件事：

- 当前到底在交付哪些 Slice，而不是“感觉上做了很多”
- 哪些需求已经形成真实能力，哪些仍然只是半闭环
- 这轮改动为什么会让用户感到“需求不清楚、功能不完全、没有审计、没有提交”

## 3. 需求基线

### 3.1 首页发现与热榜

- 首页列表默认每页 `10` 条，支持 `10 / 20 / 50 / 100`
- 首页支持中英双语切换
- 热榜必须来自真实多源同步，而不是静态 seed
- 热榜同步频率为每 `6` 小时一次，按批次增量入库
- 热度低于 `10000` 的词条不计入热榜
- 只有拿到真实直链时才允许“原文”跳转；不能把搜索页伪装成原文
- 发现文章如果仍是样例库，必须显式标识为样例，且不能伪装成实时采集结果

### 3.2 排版工作台

- 排版页必须支持真实渲染、真实保存、真实恢复
- 用户在页面上必须能看到最近草稿，并恢复到编辑区
- 封面上传应接通真实接口
- 正文资源上传与完整工作台能力仍可留在 `in_progress`，但不能误报为完成

### 3.3 导航与可见范围

- 默认主导航只暴露当前主链路和教程页
- 未完成的 P1/P2 原型不再默认暴露，避免污染用户对“系统整体可用性”的判断

## 5. 为什么上一轮会失败

### 5.1 失败点一：范围没有冻结成“明确需求基线”

虽然前面有按 Slice 推进实现，但没有把“本轮只做什么、不做什么”沉淀成单独的审计/范围文档，导致：

- 用户看到热榜已经真实了，就自然预期发现文章也应是真实采集
- 排版页虽然接了接口，但页面体验和工作台边界没有被明确说明
- 结果就是“代码有变化，但用户感知不到交付边界”

### 5.2 失败点二：审计没有形成独立产物

前面做了测试和接口验证，但没有把以下内容正式写成审计输出：

- 本轮涉及哪些 Slice
- 每个 Slice 当前是 `done` 还是 `in_progress`
- 哪些地方仍然是样例、回退或半闭环
- 哪些问题是阻塞项，哪些只是后续增强

这违反了 Harness 中“验证结果必须可追踪”的要求。

### 5.3 失败点三：提交节奏没有闭环

本轮前半段确实存在“改了、跑了、但没立即提交”的问题。原因不是 git 不可用，而是执行上没有把：

`文档回写 -> 验证 -> 审计结论 -> git commit`

当作一个不可拆分的尾门动作。

这属于执行失误，不属于用户要求不清楚。
